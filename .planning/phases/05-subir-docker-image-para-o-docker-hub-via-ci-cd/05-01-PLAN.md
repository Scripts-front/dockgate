ATENCAO CRITICA — pitfalls da pesquisa que DEVEM ser evitados:
1. `PUT /api/stacks/{id}` EXIGE `?endpointId={N}` como query param — sem ele retorna 404/500
2. Usar `RepullImageAndRedeploy: true` (NAO `pullImage: true` — campo legado)
3. Usar `jq -Rs .` para escapar o StackFileContent (YAML multi-linha) como JSON string valida
4. NAO usar `echo $SECRET` em nenhum ponto (mascaramento automatico do GitHub nao e garantido em todos os contextos)

**Secrets necessarios (referenciar exatamente com estes nomes):**
- `DOCKERHUB_USERNAME` — usuario Docker Hub (D-03)
- `DOCKERHUB_TOKEN` — Access Token Docker Hub, NAO senha (D-03)
- `PORTAINER_URL` — ex: `https://portainer.meudominio.com` (D-09)
- `PORTAINER_USERNAME` — usuario Portainer (D-06)
- `PORTAINER_PASSWORD` — senha Portainer (D-06)
- `PORTAINER_STACK_ID` — ID numerico da stack no Portainer (D-09)
- `PORTAINER_ENDPOINT_ID` — ID do ambiente Docker no Portainer (normalmente `1`; obrigatorio para o PUT funcionar — pesquisa open question resolvida aqui)

Conteudo exato do arquivo:

```yaml
name: Deploy

on:
  push:
    branches:
      - master

jobs:
  build-push:
    name: Build and Push to Docker Hub
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Login to Docker Hub
        uses: docker/login-action@v4
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v4

      - name: Build and push
        uses: docker/build-push-action@v7
        with:
          context: .
          push: true
          tags: biellil/dockgate:latest
          platforms: linux/amd64

  portainer-deploy:
    name: Update Portainer Stack
    runs-on: ubuntu-latest
    needs: build-push

    steps:
      - name: Redeploy stack via Portainer API
        run: |
          # Step 1 (D-06): Authenticate — POST /api/auth -> JWT
          JWT=$(curl -fsSL -X POST "${{ secrets.PORTAINER_URL }}/api/auth" \
            -H "Content-Type: application/json" \
            -d "{\"Username\":\"${{ secrets.PORTAINER_USERNAME }}\",\"Password\":\"${{ secrets.PORTAINER_PASSWORD }}\"}" \
            | jq -r '.jwt')

          if [ -z "$JWT" ] || [ "$JWT" = "null" ]; then
            echo "ERROR: Failed to obtain JWT from Portainer"
            exit 1
          fi

          # Step 2 (D-07): Get current stack file — GET /api/stacks/{id}/file
          STACK_FILE=$(curl -fsSL \
            -H "Authorization: Bearer $JWT" \
            "${{ secrets.PORTAINER_URL }}/api/stacks/${{ secrets.PORTAINER_STACK_ID }}/file" \
            | jq -r '.StackFileContent')

          if [ -z "$STACK_FILE" ] || [ "$STACK_FILE" = "null" ]; then
            echo "ERROR: Failed to retrieve stack file from Portainer"
            exit 1
          fi

          # Step 3 (D-08): Redeploy with pull — PUT /api/stacks/{id}?endpointId={N}
          # CRITICAL: ?endpointId is mandatory — API returns 404/500 without it
          # Field is RepullImageAndRedeploy (NOT pullImage — legacy field)
          curl -fsSL -X PUT \
            -H "Authorization: Bearer $JWT" \
            -H "Content-Type: application/json" \
            "${{ secrets.PORTAINER_URL }}/api/stacks/${{ secrets.PORTAINER_STACK_ID }}?endpointId=${{ secrets.PORTAINER_ENDPOINT_ID }}" \
            -d "{\"StackFileContent\": $(echo "$STACK_FILE" | jq -Rs .), \"Env\": [], \"Prune\": false, \"RepullImageAndRedeploy\": true}"

          echo "Portainer stack redeploy triggered successfully"
```

<threat_model>
## Trust Boundaries

| Fronteira | Descricao |
|-----------|-----------|
| GitHub Actions runner -> Docker Hub | Credenciais DOCKERHUB_TOKEN transmitidas via HTTPS; runner e efemero e isolado |
| GitHub Actions runner -> Portainer API | PORTAINER_PASSWORD transmitida via HTTPS; JWT temporario obtido e descartado ao fim do job |
| GitHub Secrets storage -> workflow YAML | Secrets nunca aparecem em texto claro nos logs; GitHub os mascara automaticamente |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Information Disclosure | deploy.yml — secrets no run step | mitigate | Usar `${{ secrets.* }}` em todos os lugares; NUNCA usar `echo $PORTAINER_PASSWORD` ou similar; GitHub mascara valores de secrets registrados nos logs |
| T-05-02 | Tampering | PUT /api/stacks sem validacao de resposta | mitigate | Usar `curl -fsSL` (fail on HTTP error); validar JWT nao-nulo antes do PUT; `exit 1` explicito em falhas |
| T-05-03 | Spoofing | PORTAINER_URL com HTTP (sem TLS) | mitigate | Documentar que PORTAINER_URL DEVE usar `https://`; nao usar `curl -k` (skip TLS) em nenhum ponto do workflow |
| T-05-04 | Elevation of Privilege | DOCKERHUB_TOKEN com escopo admin | accept | Risco residual: usuario deve gerar Access Token com escopo Read/Write apenas (sem Admin) — documentado no comentario do YAML; nao ha controle tecnico no workflow para forcar isso |
| T-05-05 | Information Disclosure | StackFileContent no JSON inline | mitigate | `jq -Rs .` escapa corretamente o conteudo YAML; o conteudo do compose nao contem secrets (apenas referencias a variaveis de ambiente preenchidas no Portainer) |
| T-05-06 | Denial of Service | Portainer JWT expirado | accept | JWT Portainer tem duracao tipica de 8h; workflow completo leva < 5 minutos; risco pratico inexistente para este pipeline |
| T-05-07 | Tampering | `linux/amd64` forcado sem verificacao de integridade | accept | Imagem publicada e publica (`biellil/dockgate`); sem provenance signing por ora — aceitavel para repositorio interno |
</threat_model>
