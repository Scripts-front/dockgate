Adicionar um segundo job `typecheck:` ao `.github/workflows/lint.yml`, paralelo ao job `lint:` existente. O arquivo final DEVE ter esta estrutura exata:

```yaml
name: Lint

on:
  push:
    branches:
      - master
  pull_request:
    branches:
      - master

jobs:
  lint:
    name: ESLint
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Bun
        uses: oven-sh/setup-bun@v2
        with:
          bun-version: latest

      - name: Install dependencies
        run: bun install --frozen-lockfile

      - name: Run ESLint
        run: bun run lint

  typecheck:
    name: Typecheck
    runs-on: ubuntu-latest

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Bun
        uses: oven-sh/setup-bun@v2
        with:
          bun-version: latest

      - name: Install dependencies
        run: bun install --frozen-lockfile

      - name: Run Typecheck
        run: bun run typecheck
```

CRÍTICO: O campo `name: Typecheck` do job é o contexto exato que o GitHub usa para status checks. Se este nome estiver errado, a branch protection ficará pendente indefinidamente. Não alterar o nome do job `lint` existente (mantém `name: ESLint` que já é required check registrado).

NÃO adicionar `needs: lint` no job typecheck — os dois devem rodar em paralelo, não em sequência.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Repositório público → GitHub Actions | Workflows são código — um PR malicioso poderia injetar steps perigosos |
| Contribuidor externo → master | Sem branch protection, push direto sem CI é possível |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Tampering | `.github/workflows/lint.yml` | mitigate | Branch protection (plano 02) bloqueará push direto sem checks; lint.yml com typecheck é o check que valida integridade do código |
| T-06-02 | Elevation of Privilege | GitHub Actions workflow | accept | Workflow não tem permissões write além do escopo padrão; `contents: read` é suficiente para lint e typecheck. Risco residual baixo para CI sem deploy |
| T-06-03 | Information Disclosure | SECURITY.md com email público | accept | Email de contato exposto intencionalmente — esse é o propósito do arquivo. Canal de reporte de vulnerabilidades requer contato público |
| T-06-04 | Denial of Service | CI runner tempo de execução | accept | Projeto pequeno, workflows simples (< 2 min). Sem rate limiting necessário no plano atual |
</threat_model>
