Executar o seguinte comando `gh api` para configurar branch protection no master. Este é o payload exato testado durante a pesquisa contra a API real do GitHub:

```bash
gh api repos/Scripts-front/dockgate/branches/master/protection \
  --method PUT \
  --header "Accept: application/vnd.github+json" \
  --input - <<'EOF'
{
  "required_status_checks": {
    "strict": false,
    "checks": [
      {"context": "ESLint"},
      {"context": "Typecheck"}
    ]
  },
  "enforce_admins": false,
  "required_pull_request_reviews": null,
  "restrictions": null,
  "allow_force_pushes": false,
  "allow_deletions": false
}
EOF
```

PARÂMETROS CRÍTICOS — não alterar:
- `"restrictions": null` — sem restrições de push; push direto continua permitido (D-01, D-04)
- `"enforce_admins": false` — Actions bot não é afetado; deploy.yml continua funcionando (D-04)
- `"allow_force_pushes": false` — push forçado bloqueado (D-03)
- `"strict": false` — não exige branch atualizada antes de merge (irrelevante sem PRs obrigatórias)
- `"required_pull_request_reviews": null` — sem exigência de review humana (D-01)
- `"checks"` usa array de objetos `{context}` — NÃO usar o campo `contexts` (deprecated)
- Os valores `"ESLint"` e `"Typecheck"` devem bater EXATAMENTE com os `name:` dos jobs no lint.yml

Após executar o PUT, verificar o resultado:
```bash
gh api repos/Scripts-front/dockgate/branches/master/protection \
  --jq '{
    force_pushes_blocked: .allow_force_pushes,
    required_checks: [.required_status_checks.checks[].context],
    restrictions: .restrictions,
    enforce_admins: .enforce_admins.enabled
  }'
```

Resultado esperado:
```json
{
  "force_pushes_blocked": false,
  "required_checks": ["ESLint", "Typecheck"],
  "restrictions": null,
  "enforce_admins": false
}
```

NOTA: `"force_pushes_blocked": false` = o campo `allow_force_pushes` retorna `false` na API, o que significa que force push está BLOQUEADO (allow=false = não permitido).

Se o comando falhar com 403: executar `gh auth status` para verificar scopes. Token precisa de `repo` e `workflow`. Fallback: configurar manualmente em https://github.com/Scripts-front/dockgate/settings/branches e documentar no SUMMARY.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| gh CLI → GitHub API | Token com escopo `repo` e `workflow` — acesso admin ao repositório |
| Branch protection rule → push direto | Regra deve bloquear force push mas não push direto normal |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-05 | Tampering | Branch protection rule | mitigate | `enforce_admins: false` evita que a própria proteção quebre o CI; `restrictions: null` preserva push direto para Actions sem criar buraco de segurança (repositório público não tem dados sensíveis) |
| T-06-06 | Elevation of Privilege | gh CLI token (repo + workflow scope) | accept | Token utilizado apenas para configuração one-time da branch protection; não armazenado em código ou logs. Risco residual: token com scope amplo — ok para repositório público |
| T-06-07 | Spoofing | Status check context name | mitigate | Os contextos "ESLint" e "Typecheck" são verificados via API antes de serem registrados (task 1 checkpoint). Nomes errados seriam detectados antes de aplicar a regra |
| T-06-08 | Denial of Service | Branch protection bloqueando deploy.yml | mitigate | `enforce_admins: false` + `restrictions: null` garantem que Actions bot continua operando. Verificado via `gh api` na pesquisa |
</threat_model>
