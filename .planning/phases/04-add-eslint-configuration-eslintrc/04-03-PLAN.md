Prerequisite check: verify `bun run lint` exits 0 before proceeding. If it fails, Plan 01 is incomplete — stop and report.

Install husky and lint-staged as dev dependencies:

```
bun add -d husky lint-staged
```

Initialize husky:

```
bunx husky init
```

This creates `.husky/pre-commit` with a placeholder. Replace its content with:

```sh
#!/usr/bin/env sh
. "$(dirname -- "$0")/_/husky.sh"

npx lint-staged
```

Note: `bunx lint-staged` may also work; prefer `npx lint-staged` for compatibility with the husky hook environment where Bun's PATH injection may not be active.

Add the `prepare` script to package.json (required for husky to install hooks on `bun install`):

```json
"prepare": "husky"
```

Add the `lint-staged` configuration key to package.json at the top level (not inside scripts). The configuration must restrict linting to staged `.ts` files within `src/` only:

```json
"lint-staged": {
  "src/**/*.ts": "eslint --no-ignore"
}
```

Note on `--no-ignore`: lint-staged passes absolute file paths to ESLint. Without `--no-ignore`, ESLint may skip files that don't match the relative glob in eslint.config.js. The `--no-ignore` flag ensures ESLint processes the file regardless of ignore rules when lint-staged has already filtered to the correct set.

Final package.json scripts section must include `prepare`:

```json
"scripts": {
  "start": "bun run src/index.ts",
  "dev": "bun --hot src/index.ts",
  "typecheck": "tsc --noEmit",
  "lint": "eslint src",
  "prepare": "husky"
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer workstation → git commit | Pre-commit hook intercepts staged files before commit lands |
| lint-staged → ESLint | lint-staged passes absolute paths; --no-ignore ensures ESLint processes them |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-07 | Tampering | .husky/pre-commit | accept | Hook can be bypassed with `git commit --no-verify`. This is a developer convenience tool, not a security boundary. CI (Plan 02 workflow) provides the authoritative enforcement |
| T-04-08 | Denial of Service | lint-staged blocking commits | accept | Intentional design — lint errors block commits. Developer fixes violations before committing |
| T-04-09 | Tampering | lint-staged config glob | mitigate | Glob is scoped to `src/**/*.ts` only. Files outside src/ are never linted by the hook, preventing false positives on config files |
</threat_model>
